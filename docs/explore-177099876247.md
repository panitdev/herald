# Comprehensive Codebase Exploration Summary

## 1. Design System

### Colors & Tokens

The project uses **CSS custom properties** defined in OKLCH color space for a perceptually uniform design system. There are **two globals.css files** with different palettes:

- **`ui/app/globals.css`** — Primary palette with soft indigo primary (`oklch(0.55 0.18 258)`), neutral backgrounds, and `--radius: 0.75rem`
- **`ui/styles/globals.css`** — Alternative neutral palette with darker primary (`oklch(0.205 0 0)`), `--radius: 0.625rem`

Both define the full shadcn/ui token set:

- `--background`, `--foreground`, `--card`, `--popover`
- `--primary`, `--secondary`, `--accent`, `--muted`, `--destructive`
- `--border`, `--input`, `--ring`
- `--chart-1` through `--chart-5`
- Full `--sidebar-*` token family

### Typography

- **Font**: `Geist` (sans) and `Geist Mono` (mono) — loaded via `next/font/google`
- **Size scale**: Tailwind v4's fluid scale (`text-sm`, `text-[13px]`, `text-[11px]`, `text-xs`)
- **Tracking**: `tracking-tight` used on headings

### Spacing

- Tailwind v4 spacing scale
- Border radius system: `--radius-sm`, `--radius-md`, `--radius-lg=0.75rem`, `--radius-xl`
- Density modes supported: `comfortable | cozy | compact` (applied via `data-density` attribute)
- Custom scrollbar utility: `.scrollbar-thin`

### Accent System

Dynamic accent colors defined in `lib/settings-store.tsx` — 5 presets (indigo, emerald, rose, amber, slate) that override CSS custom properties at runtime.

## 5. Routing Structure

### Next.js App Router

Very simple route structure:

| Route             | File                       | Purpose                  |
| ----------------- | -------------------------- | ------------------------ |
| `/**` (catch-all) | `app/[[...slug]]/page.tsx` | Renders the email client |

### Client-Side URL Management

Since the app is a single-page email client, routing is handled manually via the History API:

**`ui/lib/mail-navigation.ts`**:

```ts
navigateToMail(id)      → /mail/${id}
navigateToMailbox(box)  → /${mailbox} (inbox, sent, drafts, etc.)
navigateToRoot()        → /
getMailIdFromUrl()      → parses /mail/:id
getMailboxFromUrl()     → parses /:mailbox
```

**Hook**: `ui/hooks/use-mail-navigation.ts` — Synchronizes React state with URL paths. No `next/navigation` router used for internal mail routing.

### API Routes (backend)

`apps/api/src/index.ts` — Hono server on Cloudflare Workers:

| Endpoint                      | Auth       | Description            |
| ----------------------------- | ---------- | ---------------------- |
| `POST /api/auth/register`     | Public     | User registration      |
| `POST /api/auth/login`        | Public     | User login             |
| `POST /api/auth/refresh`      | Public     | Token refresh          |
| `POST /api/auth/logout`       | Public     | Logout                 |
| `GET /api/messages`           | Bearer JWT | List messages          |
| `GET /api/messages/:id`       | Bearer JWT | Get single message     |
| `POST /api/messages/:id/read` | Bearer JWT | Mark read              |
| `GET /api/messages/:id/raw`   | Bearer JWT | Raw email content (R2) |
| `GET /api/ws`                 | Bearer JWT | WebSocket realtime     |
| `GET/POST /api/mailboxes`     | Bearer JWT | Mailbox CRUD           |

## 6. Auth-Related Code

### Backend Auth (`apps/api/src/auth.ts`)

Custom JWT implementation using WebCrypto API (no external JWT library):

- **JWT signing**: HMAC-SHA256 (`signHMAC` / `verifyHMAC`)
- **Access token expiry**: 15 minutes
- **Refresh token expiry**: 7 days
- **Password hashing**: PBKDF2 with 100,000 iterations, 16-byte random salt
- **Token hash**: SHA-256 for refresh token storage
- **Secret**: Hardcoded as `"herald-jwt-secret-change-in-production"` ⚠️

**Middleware**: `authMiddleware` extracts `Bearer <token>`, verifies signature + expiration, sets `c.set("user", { id, email })`

### Database Schema (inferred from SQL)

- `users` table: `id`, `email`, `password_hash`
- `refresh_tokens` table: `id`, `user_id`, `token_hash`, `expires_at`
- `mailboxes` table: `id`, `address`, `user_id`
- `messages` table: `id`, `mailbox_id`, `thread_id`, `from_addr`, `subject`, `preview`, `r2_raw_key`, `received_at`, `read_at`

### Frontend Auth UI

**No dedicated login/register pages exist yet** in the UI. The `profile-menu.tsx` has:

- "Session" section with a "Sign out" button (triggers `toast.success("Signed out")` — currently just a UI placeholder)
- Account/Settings items open the settings dialog

The frontend currently works with **mock data** (`lib/mock-emails.ts`) and the API auth endpoints are implemented but not yet wired to the UI.
